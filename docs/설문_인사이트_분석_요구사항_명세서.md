# 설문 인사이트 분석 시스템 — 요구사항 명세서

**문서 버전:** v0.1 (초안)
**목적:** 기존 설문 시스템에 부가될 "감사 가능한 AI 인사이트 분석 + 대화형 드릴다운" 서브시스템의 요구사항을 정의한다. 본 문서는 아키텍처 설계의 입력으로 사용된다.
**작성 기준:** 본 명세서는 *무엇을(What)* 만들지를 규정하며, *어떻게(How)* 구현할지는 아키텍처 설계 단계에서 결정한다. 단, 신뢰성·추적성 요구는 구조에 직접 영향을 주므로 데이터 모델 수준까지 기술한다.

---

## 1. 개요

### 1.1 배경
전사 그룹사(약 12만 명)를 대상으로 연 2회 실시하는 회사/조직 문화·역량 평가 설문이 운영 중이다. 객관식 응답은 통계로 처리되나, 다량의 **자유 응답(주관식)** 은 충분히 활용되지 못하고 있다. 본 시스템은 이 자유 응답을 분석해 조직 단위 인사이트를 생성하고, 그 결과가 KPI로 쓰이는 중요 리포트인 만큼 **판단 근거와 추론 과정을 추적·검증·질의할 수 있게** 한다.

### 1.2 시스템 범위
- 대상 데이터: 전사 문화·역량 설문의 자유 응답 (객관식 통계는 보조 입력으로 활용 가능)
- 핵심 산출물: (1) 조직별 인사이트 리포트, (2) 각 인사이트의 근거·추론 트레이스, (3) 리포트에 대한 대화형 질의응답
- 본 명세서는 분석 서브시스템에 한정하며, 설문 수집/배포 기능 자체는 기존 시스템을 따른다.

### 1.3 용어 정의
| 용어 | 정의 |
|------|------|
| 응답(Response) | 한 응답자가 특정 설문 문항에 작성한 개별 자유 응답. 익명화된 식별자를 가짐 |
| 주제(Theme) | 다수 응답을 군집화해 도출한 의미 단위(예: "업무량 과중") |
| 인사이트(Insight) | 주제·통계·추세로부터 도출된 판단(예: "A본부 업무량 우려 상승") |
| 근거(Evidence) | 인사이트를 뒷받침하는 실제 응답, 통계, 인용문 |
| 추론 트레이스(Reasoning Trace) | 인사이트가 도출된 과정을 구조화·재현 가능하게 기록한 것 |
| 출처 그래프(Provenance Graph) | 응답→주제→인사이트→리포트 문장을 잇는 링크 구조 |
| 회차(Cycle) | 연 2회 실시되는 설문 단위 |
| k-익명성 | 최소 응답 인원 임계치 미만 조직의 결과를 노출하지 않는 익명성 보호 규칙 |

### 1.4 핵심 원칙
- **P1. 출처 추적 우선:** 모든 산출물은 그 출처를 끊김 없이 추적할 수 있어야 한다.
- **P2. 충실한 추론(Faithfulness):** 담당자에게 보이는 "이유"는 모델의 사후 합리화(raw chain-of-thought)가 아니라, 실제 계산값·검색된 근거·검증을 통과한 단계로부터 재구성된 것이어야 한다.
- **P3. 근거 없는 주장 금지:** 근거 검증을 통과하지 못한 주장은 리포트에 포함하지 않거나 낮은 신뢰도로 명시한다.
- **P4. 익명성 보장:** 분석·질의 어느 단계에서도 응답자 재식별이 가능해서는 안 된다.

## 2. 액터 및 시스템 컨텍스트

| 액터 | 설명 | 주요 관심사 |
|------|------|------------|
| 분석 담당자 | 리포트를 검토하고 KPI 판단을 내리는 사용자 | 근거 확인, 추론 이유 확인, 대화형 질의 |
| 조직장 | 자기 조직 리포트 수령자 | 핵심 이슈, 추세, 권장 액션 |

| 기존 설문 시스템 | 응답 데이터 및 조직 구조 제공원 | 데이터 연동 인터페이스 |

## 3. 기능 요구사항 (FR)

### 3.1 데이터 수집 및 전처리
- **FR-1** 시스템은 기존 설문 시스템으로부터 한 회차의 자유 응답과 조직 구조(계층)를 가져올 수 있어야 한다.
- **FR-2** 각 응답에는 회차·문항·조직 정보가 연결된 **익명화된 고유 식별자**가 부여되어야 한다. 식별자로부터 개인을 역추적할 수 없어야 한다.
- **FR-3** 시스템은 무성의 응답(직선 응답, 무의미 텍스트 등)을 탐지·표시할 수 있어야 하며, 분석 포함 여부를 정책으로 제어할 수 있어야 한다.
- **FR-4** (다국어 운영 시) 응답 언어를 식별하고 분석 파이프라인이 다국어를 처리할 수 있어야 한다. → §9 미정 사항 참조.

### 3.2 주제 분석
- **FR-5** 시스템은 자유 응답을 임베딩 기반으로 군집화하여 주제를 도출해야 한다.
- **FR-6** 각 주제에는 사람이 이해할 수 있는 라벨과 대표 응답(인용)이 부여되어야 한다.
- **FR-7** 각 응답은 어떤 주제(들)에 배정되었는지 추적 가능해야 한다(FR-15 연계).
- **FR-8** 시스템은 주제별 감정/측면(긍·부정·중립, 가능하면 측면별 감정)을 산출해야 한다.

### 3.3 인사이트 생성
- **FR-9** 시스템은 주제·통계·회차 추세로부터 조직 단위 인사이트를 생성해야 한다.
- **FR-10** 각 인사이트에는 **신뢰도(confidence)** 가 부여되어야 한다. 신뢰도는 최소한 표본 크기, 감정 일관성, 근거성 점수를 반영해야 한다.
- **FR-11** 담당자는 신뢰도를 기준으로 "확실한 인사이트"와 "잠정적 신호"를 구분할 수 있어야 한다.

### 3.4 출처 추적 (Provenance)
- **FR-12** 시스템은 다음 링크를 모두 보존해야 한다: 응답 → 주제 → 인사이트 → 리포트 문장.
- **FR-13** 담당자는 리포트의 **임의의 문장에서 그 문장을 뒷받침하는 인사이트·주제·실제 응답까지 단계적으로 펼쳐볼 수 있어야** 한다.
- **FR-14** 임의의 인사이트에 대해 그것을 만든 근거(응답 ID 목록, 통계, 인용)를 조회할 수 있어야 한다.
- **FR-15** 출처 링크는 다단계 처리(군집화·요약·작성)를 거치는 동안 끊기지 않아야 한다.

### 3.5 추론 트레이스
- **FR-16** 각 인사이트는 **구조화된 추론 트레이스**를 가져야 한다. 트레이스는 사용된 통계값(빈도·비율·전년 대비 변화), 검색·인용된 근거, 적용된 판단 규칙을 포함한다.
- **FR-17** 추론 트레이스는 재현 가능해야 한다(동일 입력·동일 정책에서 동일한 근거·집계로 설명 가능).
- **FR-18** 시스템은 모델의 원시 사고과정(raw CoT)을 판단 근거로 제시해서는 안 된다(원칙 P2).
- **FR-19** 담당자는 인사이트별 추론 트레이스를 사람이 읽을 수 있는 형태로 확인할 수 있어야 한다.

### 3.6 근거 검증
- **FR-20** 생성된 모든 주장은 인용된 근거가 실제로 그 주장을 뒷받침하는지 **검증 단계**를 거쳐야 한다(예: 함의/근거성 평가).
- **FR-21** 검증을 통과하지 못한 주장은 리포트에서 제외하거나 명시적으로 낮은 신뢰도/경고와 함께 표시해야 한다(원칙 P3).
- **FR-22** 검증 실패 시 시스템은 재작성·재검증을 시도하는 반복 루프를 수행할 수 있어야 한다(반복 횟수 상한 설정 가능).

### 3.7 조직 계층 롤업 및 리포트 생성
- **FR-23** 시스템은 조직 계층(팀 → 본부 → 그룹사)을 따라 인사이트를 집계·롤업해야 한다.
- **FR-24** 시스템은 조직별 맞춤 리포트(핵심 테마, 전년 대비 추세, 위험 신호, 권장 액션)를 생성해야 한다.
- **FR-25** 리포트의 각 문장은 하나 이상의 인사이트(주장)에 매핑되어야 한다(FR-12 연계).
- **FR-26** 권장 액션은 근거가 된 인사이트와 연결되어야 한다.

### 3.8 회차 간 추세 비교
- **FR-27** 시스템은 회차 간 주제를 비교해 추세(상승/하락/신규/소멸)를 산출해야 한다.
- **FR-28** 추세 비교가 유효하려면 회차 간 **주제 분류 체계의 일관성**이 유지되어야 한다. 시스템은 이전 회차 주제 체계를 참조해 매핑하는 메커니즘을 가져야 한다.

### 3.9 대화형 드릴다운 (질의응답)
- **FR-29** 담당자는 리포트·인사이트에 대해 자연어로 질의하고 답을 얻을 수 있어야 한다(예: "X팀 사기가 왜 떨어졌다고 봤어?").
- **FR-30** 질의응답 에이전트는 **출처 그래프를 조회하는 도구**를 통해서만 답해야 하며, 검색된 근거에 묶이지 않은 자유 생성 답변을 해서는 안 된다.
- **FR-31** 최소한 다음 조회 기능이 제공되어야 한다: 인사이트별 근거 조회, 조건별 응답 조회, 주제·조직·회차별 통계 조회, 회차 간 비교.
- **FR-32** 모든 답변에는 근거(실제 응답 인용/통계)가 함께 제시되어야 한다.
- **FR-33** 담당자는 답변에서 더 깊이 파고들어 익명화된 원문 코멘트까지 열람할 수 있어야 한다(FR-36 제약 준수).
- **FR-34** 답변 생성 직전 충실성 검증을 거쳐, 근거와 불일치하는 답변은 차단·수정되어야 한다.
- **FR-35** 질의응답은 대화 맥락(직전 질문)을 유지할 수 있어야 한다.

### 3.10 프라이버시 보호
- **FR-36** 시스템은 **k-익명성 임계치**(예: 최소 응답 인원 미만 조직)를 적용하여, 임계치 미만 조직의 인사이트·원문은 노출하지 않아야 한다. 임계치는 설정 가능해야 한다.
- **FR-37** 원문 인용 노출 시에도 개인 식별 정보(이름, 직책 등 재식별 단서)는 마스킹/제거되어야 한다.
- **FR-38** 접근 권한에 따라 열람 가능한 조직 범위가 제한되어야 한다(조직장은 자기 조직 범위).

## 4. 비기능 요구사항 (NFR)

### 4.1 신뢰성·설명가능성
- **NFR-1** 리포트의 모든 핵심 주장은 근거까지 추적 가능해야 한다(추적 불가 주장 0건이 목표).
- **NFR-2** 추론 트레이스는 감사(audit) 가능해야 한다.

### 4.2 프라이버시·보안
- **NFR-3** 어떠한 분석·질의 경로로도 응답자 재식별이 불가능해야 한다.
- **NFR-4** 데이터 접근은 역할 기반 권한 제어를 따라야 한다.
- **NFR-5** 데이터 보관·처리는 사내 데이터 거버넌스 정책을 준수해야 한다. → §9 확인 필요.

### 4.3 규모·성능
- **NFR-6** 한 회차 약 12만 명 규모의 자유 응답(복수 문항 가능)을 처리할 수 있어야 한다.
- **NFR-7** 분석은 배치 처리이며, 1회차 분석이 운영상 허용 시간 내 완료되어야 한다. → 목표 시간 §9 확인.
- **NFR-8** 대화형 질의응답은 대화형으로 허용되는 응답 지연 내에 결과를 반환해야 한다.

### 4.4 비용
- **NFR-9** 대량 응답을 고비용 모델에 일괄 투입하지 않고, 단계적 처리(임베딩·군집화 → 저비용 1차 → 고비용 핵심 요약)로 비용을 관리해야 한다.
- **NFR-10** 시스템은 회차당 처리 비용을 측정·리포팅할 수 있어야 한다.

### 4.5 안정성·운영성
- **NFR-11** 장시간 배치는 중단 후 재개(체크포인트)가 가능해야 한다.
- **NFR-12** 단계별 실패에 대해 재시도·타임아웃 처리가 있어야 한다.
- **NFR-13** 파이프라인의 각 단계와 에이전트 의사결정은 추적(tracing)·관찰 가능해야 한다.

### 4.6 확장성
- **NFR-14** 개별 구성원 자율 설문 등 다른 설문 유형으로 분석 대상을 확장할 수 있는 구조여야 한다(향후).

## 5. AI/ML 요구사항

- **AIR-1** 주제 도출: 임베딩 + 군집화(라벨 생성 포함) 방식을 사용한다.
- **AIR-2** 생성 단계는 구조화 출력(주장·근거 ID·신뢰도 등)을 강제한다.
- **AIR-3** 근거 검증은 자동 평가(함의/근거성 판정)를 사용하며, 검증→재작성 반복 루프를 지원한다.
- **AIR-4** 대화형 질의응답은 도구 사용 기반의 제약된 검색 증강(RAG) 방식으로 동작한다.
- **AIR-5** 가드레일: 답변·리포트는 근거 범위를 벗어난 주장, 재식별 위험 내용을 생성하지 않도록 제어되어야 한다.
- **AIR-6** 모델·프레임워크 선택은 아키텍처 단계에서 확정하되, 상태 그래프·반복 루프·체크포인트·도구 사용을 지원하는 오케스트레이션을 전제로 한다.

### 5.1 평가 요구사항
- **AIR-7** 인사이트 품질, 인용/근거 정확도, 대화 답변 충실성 세 가지를 각각 평가할 수 있어야 한다.
- **AIR-8** 자동 평가 지표(예: 충실성, 근거 정밀도/재현율, 답변 적합성)와 사람 검수(스팟체크) 하니스를 함께 운영한다.
- **AIR-9** 평가용 기준 데이터셋(대표 응답·기대 결과)을 구축·유지한다.

## 6. 데이터 모델 (출처 그래프 — 개념 수준)

아키텍처 설계 시 다음 엔티티와 관계가 보존되어야 한다(구현 형태는 자유).

| 엔티티 | 핵심 속성 | 관계 |
|--------|-----------|------|
| Response | response_id(익명), cycle_id, question_id, org_id, text, language, quality_flag | → Theme (n:m) |
| Theme | theme_id, label, cycle_id, representative_response_ids, sentiment | ← Response, → Insight, ↔ 이전 회차 Theme |
| Insight | insight_id, statement, org_id, confidence, evidence_refs, trend | ← Theme, → ReportSentence |
| ReasoningTrace | trace_id, insight_id, stats, cited_evidence, applied_rules, verification_result | ↔ Insight (1:1) |
| ReportSentence | sentence_id, text, org_id, claim_refs | ← Insight |
| OrgUnit | org_id, parent_org_id, level | 계층 구조 |
| Cycle | cycle_id, period | |

핵심 불변식: 모든 ReportSentence는 1개 이상의 Insight에 연결되고, 모든 Insight는 ReasoningTrace와 1개 이상의 Evidence(Response/통계)에 연결된다.

## 9. 미정 사항 / 결정 필요

아키텍처 설계 전에 확정이 필요한 항목.

- **Q-1** 그룹사가 다국어 환경인가? (응답 언어 종류, 다국어 분석 필요 여부 — FR-4/NFR 영향)
- **Q-2** 회차당 자유 응답 총량과 문항 수 추정치는? (NFR-6/7 비용·성능 설계 영향)
- **Q-3** 1회차 분석 허용 완료 시간 목표는? (NFR-7)
- **Q-4** k-익명성 최소 임계치 값은? (FR-36, 통상 5 등 — 사내 정책 확인)
- **Q-5** 사용 가능한 LLM/모델과 데이터 반출 정책(외부 API 사용 가능 여부, 온프레미스 필요 여부)은? (AIR-6/NFR-5)
- **Q-6** 신뢰도(confidence) 등급 기준과 "리포트 포함 최소 임계치"는?
- **Q-7** 권장 액션 생성 범위 — AI가 제안까지 할지, 인사이트까지만 제공할지?
- **Q-8** 기존 설문 시스템과의 연동 방식(API/DB/배치 파일)과 인증 체계는?
- **Q-9** 8주 1차 구현 범위 — 전체 vs 축소판(아래 §10).

## 10. 1차(8주) 범위 권고 및 범위 외

### 10.1 1차 권고 범위
- 단일 회차, 단일 조직 레벨, 표본 데이터 기준.
- 핵심: 출처 추적(FR-12~15), 근거 검증(FR-20~22), 추론 트레이스(FR-16~19), 대화형 드릴다운(FR-29~34), k-익명성(FR-36)을 동작 수준으로 구현.
- 평가 하니스(AIR-7~9) 기본형 포함.

### 10.2 범위 외(향후 단계)
- 회차 간 추세 비교의 완전 자동화(FR-27/28)는 2차로 이연 가능.
- 전 조직 계층 전면 롤업(FR-23) 및 전수 12만 명 운영 규모 처리(NFR-6)는 1차 이후.
- 자율 설문 등 타 설문 유형 확장(NFR-14).
- 권장 액션 효과의 차기 회차 추적(폐루프).

## 11. 수용 기준 / 성공 지표

- **AC-1** 리포트의 임의 문장에서 근거 응답까지 단계적으로 추적 가능(추적 불가 0건).
- **AC-2** 근거 검증을 통과하지 못한 주장이 리포트에 무표시로 포함되지 않음.
- **AC-3** 대화형 질의의 답변이 모두 근거 인용을 동반하며, 충실성 평가 기준치 이상.
- **AC-4** k-익명성 임계치 미만 조직의 인사이트·원문이 어떤 경로로도 노출되지 않음.
- **AC-5** 평가 데이터셋 기준으로 인사이트 품질·인용 정확도·답변 충실성 지표가 합의된 기준치 이상.
- **AC-6** 회차당 처리 비용이 측정·리포팅되며 목표 예산 내.
